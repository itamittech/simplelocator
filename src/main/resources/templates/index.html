        Query Flow

        What PostgreSQL does internally for each index

        GiST R-tree Path
        1. Receive ST_DWithin(location, center, 8047m)
        2. Walk R-tree root → skip branches whose MBR > 8 km from center
        3. At leaf nodes → exact distance check
        4. Return matched rows ordered by distance
        SP-GiST Quadtree Path
        1. Build envelope: ST_Expand(center, 0.09°)
        2. && → SP-GiST descends only quadrants overlapping the box
        3. Collect candidate rows from matching leaf quadrants
        4. ST_DWithin exact filter → discard box corners
        Geohash B-tree Path
        1. Encode center → geohash precision-4 string
        2. Compute 8 neighboring cells → 9 prefix strings
        3. B-tree scan: LIKE 'dr5q%' OR LIKE 'dr5r%' OR…
        4. ST_DWithin exact filter → remove rectangle corners

        Menu Search — GIN Index + JSONB

        One query — three PostgreSQL index types working together: GiST for location, GIN for full-text, GIN for structured JSON

        GIN
        Inverted Index
        Full-Text Search
        pasta→row 19 (Battello)
        pizza→rows 1, 11, 12, 19
        miso→row 8 (Nobu)
        curry→rows 16, 18
        dumpling→rows 17 (Nan Xiang)
        Lexeme → posting list (row IDs)
        • Tokenizes text into lexemes (stemmed words) stored as a tsvector
        • GIN maps each lexeme to a posting list of matching row IDs
        • Query: menu_search_vector @@ plainto_tsquery('pasta')
        • Covers: restaurant name + cuisine + all menu item names + descriptions
        tsvector column
        -- Stored at migration time:
menu_search_vector =
  to_tsvector('english',
    name || cuisine ||
    item_names || item_descriptions
  )

-- Queried via GIN:
WHERE menu_search_vector
  @@ plainto_tsquery('english', ?)
        JSONB
        Binary JSON
        GIN Containment Index
        menu column (JSONB):
        {
  "items": [
    { "name": "Veggie Ramen",
      "price": 19.00,
      "dietary": ["vegetarian","vegan"] }
  ],
  "dietary_options":
    ["vegetarian","vegan"],
  "price_range": "$$"
}
        Structured document — GIN indexes every key + value path
        • JSONB stores documents as binary decomposed key-value pairs — unlike JSON, it's indexable
        • GIN on JSONB indexes every path and value in the document
        • @> containment check: does the left document contain the right?
        • Works like a NoSQL document filter inside PostgreSQL
        GIN JSONB containment
        -- GIN index created on menu column:
CREATE INDEX ON restaurants
  USING GIN(menu);

-- Dietary filter via @> (contains):
WHERE menu @> '{
  "dietary_options": ["vegetarian"]
}'::jsonb

        PostgreSQL intersects all three index scans
        • PostgreSQL's bitmap index scan intersects results from all three indexes

        Combined SQL Pattern
        WHERE
  ST_DWithin(location_gist, ?, 8047)
  -- ① GiST: prune by location

  AND menu_search_vector

  -- ② GIN/tsvector: prune by text

  AND menu @> ?::jsonb
  -- ③ GIN/JSONB: prune by dietary

ORDER BY
  ts_rank(...) DESC,  -- relevance first
  dist_meters ASC     -- then distance

        Rider Mode — Redis GEO

        A delivery rider drives through Manhattan — Redis finds nearby restaurants in real-time using native geo commands

        Redis
        GEO Commands
        Sorted Set (52-bit geohash score)
        KEY: restaurants:geo Sorted Set
        score (geohash52)member
        3.29720…×10¹⁴"1" → Joe's Pizza
        3.29721…×10¹⁴"5" → Nobu
        3.29724…×10¹⁴"12" → Le Bernardin
        ……
        Nearby IDs share close score values — O(log N) radius scan
        Redis encodes (lng, lat) into a single 52-bit integer score
        • All 23 restaurants loaded via GEOADD on app startup
        • GEOSEARCH (Redis 6.2+) replaces deprecated GEORADIUS — same O(log N) scan
        • Sub-millisecond for small datasets; ideal for high-frequency location updates
        • Works alongside PostGIS — Redis for fast proximity, PostGIS for geofencing
        Redis Commands
        # Load (once at startup)
GEOADD restaurants:geo lng lat "id" …

# Query on every rider location update
GEOSEARCH restaurants:geo
  FROMLONLAT lng lat
  BYRADIUS 2 mi
  ASC WITHCOORD WITHDIST COUNT 20
        Redis + PostGIS Hybrid Query Flow
        1. App starts → GEOADD loads all restaurants into a Redis sorted set
        2. Rider moves → POST /api/rider/update with lat/lng
        3. Redis GEOSEARCH FROMLONLAT … BYRADIUS 2 mi → list of nearby IDs
        4. PostGIS ST_Within(point, zone.boundary) → current delivery zone
        5. Zone change or close restaurant detected → SSE event pushed to browser
        6. Browser receives event via EventSource — no polling needed
        RedisFast proximity — all in RAM, no SQL
        PostGISGeofencing — polygon containment (ST_Within)
